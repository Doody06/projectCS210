We want to handle exceptions only where they are actually needed (such as file reading and parsing), and avoid wrapping everything blindly in try/catch blocks.

We want to use OOP whenever it is possible and keep classes small, simple, and focused on a single responsibility.

We MUST avoid using the Java Collections Framework completely and implement all required data structures manually from scratch.

We want to strictly follow the required class names, attributes, and method signatures without modifying the given API.

We want each data structure to be independent and responsible only for its own logic without mixing responsibilities between classes. (High Modularity and Low Coupling!)

We want all operations to respect their expected time complexity (e.g., AVL Tree operations in O(log n), Queue and Stack operations in O(1)).

We want to avoid duplicating logic and instead use helper methods when the same functionality is needed in multiple places.

We want to ensure that console input follows the exact required format and handle invalid input without crashing the program.

We want every operation to correctly update all related data structures to maintain system consistency.

We want the undo operation to fully and correctly reverse the last action without leaving partial changes.

We want to keep methods short, clear, and easy to debug instead of writing large, complex blocks of code. (except AVL...)

We want to use clear and meaningful names for variables, methods, and classes to make the code easier to understand and explain.

We want to avoid hardcoding values and rely only on input from files or the console.

We want to validate file input before processing to ensure it follows the correct format.

We want to maintain a consistent coding style across all classes (naming, formatting, and structure).
